# 线上变更管理军规

# 1、目的

变更上线军规的目的是确保变更过程高效、安全、可控，最大限度地降低服务中断风险，保障系统稳定运行和用户体验。

# 2、适用范围

本规范适用于换电系统全体人员，适用于所有生产研发系统、第三方运维系统的线上变更。

**包括：**

- **代码发布**：服务端代码发布、客户端发布、WEB、H5页面、离线任务、脚本等;
- **配置变更**：apollo配置、实验配置、投放配置、接入层配置（disf、域名、路由转发）等配置系统中线上环境操作或同步;
- **数据变更**：Mysql、Redis、Fusion、ES、Gift等存储服务线上环境操作;
- **基础组件变更**：服务器、网络设备、存储等硬件的增减、迁移或配置变动；
- **流量变化**：压测等会对线上流量产生变化的操作;

**不包括：**

- 对未上线的服务、配置的变更，如：未承接正式流量（且变更不影响正式流量）的服务
- 测试环境(无生产流量)的程序、配置调试。

# 3、军规要求

### 3.1、设计阶段

**可监控、可灰度、可回滚** — 重要功能设计必须保证**小流量开关、快速回滚方案、部署方案、上线观察项和监控方案完备**；**变更内容需要与技术方案文档保持一致。**

### 3.2、测试阶段

测试方案中必须覆盖正反逻辑，必须覆盖兜底逻辑，必须包含异常和异常边界测试用例；严禁在SIM环境进行压测；

### 3.3、准入阶段

测试准入不通过必须打回；

### 3.4、准出阶段

准出不通过禁止上线；研发自测上线需遵循自测上线流程；准出之后代码不允许更改；

### 3.5、上线阶段

**提前通报要记得** — 上线前必须提前通报关联方，严禁私自上线；

**高峰窗口要避免** — 高峰期严禁上线和变更，特殊情况需要经由平台或邮件进行紧急审批；

**变更申请要规范** — 上线前需明确变更内容、发布方案、发布步骤、变更时间、验证方法、回滚止损方案。

**分级发布要遵守** — 所有线上变更不可以一次性作用到线上 全部流量，至少分级为3步：小流量-->中流量-->全量。第一次停留至少5分钟，全量前停留至少10分钟；每个阶段严格按检查项检查。

**变更步骤要完备 — 任何线上变更都要事先准备好回滚预案，一旦操作中出现异常按预案执行回滚；**

**服务检查要执行** — 服务以及业务无异常；

**代码变更需要走checklist平台**、**研发配置需要通过配置中心**、**实验变更需要通过实验平台进行审批通过之后再进行变更。**

### 3.6、线上事故处理阶段

先通报后处置，先止损后解决；止损最高优，善后不可忘；

问题出现第一时间同步到[一般问题线上同步群](https://im.xiaojukeji.com/channel?uid=425865&token=64e40d257ccd8f776f042d8a62ac0b9d&id=1951432056499246080)，通报模板建议如下：

### 3.7、复盘阶段

所有线上P5+故障和违反军规变更导致的故障必须复盘，反思要深入、改进要落地。改进项要按时交付。

时间窗和审批人

- **标准上线窗口**
- 时间区间：工作日11:00 ~ 12:00；14:00 ~ 22:00;
- 变更审批：按照标准上线流程审批和执行，模块负责人审批
- 通报范围：部署群
- **业务封线时间**
- 时间区间：00:00-11:00,12:00-14:00,22:00-23:59; 周末全天；
- 变更审批：原则上禁止在此期间上线，特殊情况需要走紧急流程，由二级审批人审批
- 通报范围：部署群
- **假期和特殊时期**
- 时间区间：具体时间提前一周发布；
- 变更审批：原则上禁止在此期间上线，特殊情况需要走紧急流程，由一级审批人审批
- 通报范围：提前邮件通报到相关「业务/研发/运维」团队，涉及到核心链路主流程相关，还需邮件通报到星辰花团队

补充说明

- 节假日封线具体时间可能视情况调整，遵循；
- 如时间定义有重叠，则执行更高级别的审批要求；
- 部分特殊业务的封线时间不同于以上设置，需要联系sre在系统单独设置。

# 6、军规培训、宣贯计划

红线操作

**手工操作线上数据、代码、配置：**

- 直接使用 MySQL 客户端（包含脚本）读写线上主、从数据库
- 直接使用 Redis 客户端（包含脚本）读写线上Redis、Fusion 、RT 库
- 编写未经 QA 评估的脚本、命令操作线上数据
- 手工登录生产环境机器变更线上机器代码、配置
- 手工登录生产环境机器对服务进行停止、重启
- 未走压测流程（压测日历申请、提前消防群通报），私自在线上进行服务压测
- 未经过T2同意直接通过脚本修改线上数据，刷数据没有回滚机制

**上线或者发版不规范：**

- 未通知相关审批人或 T3+ 未评估情况下，擅自跳过上线审批
- 上线不做观察（比如：上线后就吃饭去了）
- 不知道本次上线的影响面，不知道观察什么监控指标
- 针对不符合预期的情况，没有回滚预案，或者预案无法执行
- 上线期间出现一级报警，继续上线
- 不通过app发布平台集成tag和aar（APP发版操作）
- 未经开关owner同学确认，擅自操作生产环境开关
- 进行线上变更，24小时内保持oncall，避免因变更出现故障无法及时响应
- 禁止提交和集成与当前发布不相干的代码(搭车行为)

**封线期间擅自操作：**

- 擅自变更线上 apollo 开关、配置
- 擅自操作线上 911 降级开关
- 擅自变更线上 ddmq 生产、消费 topic
